# Requirements — ДЗ1: Анализ требований (Кейс A: доставка еды)

## 0. Контекст и границы системы

**Кейс:** сервис доставки еды (аналог Яндекс.Еда) для **Россия + СНГ**.  
**Аудитория:** **5–20 млн MAU**.  
**Инфраструктура:** облако, бюджет ограничен.  
**Вводные от заказчика:**
- «Система должна выдерживать **обеденный пик**»
- «**Оплата — самое важное**, ничего не должно теряться»
- «Но скорость оформления критична — клиенты уходят, если ждут больше пары секунд»
- «Система должна работать быстро и не падать»

### 0.1. Роли (акторы)
- **Клиент:** ищет рестораны/блюда, оформляет и оплачивает заказ, отслеживает доставку, получает уведомления.
- **Ресторан:** управляет меню/наличием, принимает заказ, меняет статусы (готовится → готов).
- **Курьер:** принимает доставку, обновляет статусы (в пути/доставлено).
- **Оператор поддержки/модератор:** решает спорные случаи, отмены/возвраты, проверка ресторанов/контента.
- **Платёжный провайдер (PSP/эквайринг).**
- **Провайдеры уведомлений:** push / SMS.

### 0.2. In-scope (в рамках системы)
- Каталог ресторанов и меню, поиск/фильтры.
- Корзина, оформление заказа.
- Оплата заказа и фиксация статуса.
- Трекинг статуса доставки в реальном времени (статусы).
- Уведомления.
- История заказов и повтор заказа (reorder) — как часть UX (можно минимально).

## 1. Часть 1 — Функциональные требования (ФТ)


**ФТ-001:** Пользователь может просматривать и искать рестораны и блюда 
по названию, кухне, категории и фильтрам (цена, рейтинг, время доставки).

**ФТ-002:** Пользователь может добавлять блюда в корзину, изменять количество/опции и видеть итоговую стоимость (товары + доставка + скидки).  

**ФТ-003:** Пользователь может оформить заказ: выбрать адрес, комментарий; после оформления создаётся сущность заказа с уникальным идентификатором и статусом.  

**ФТ-004:** Пользователь может оплатить заказ онлайн; система должна **надёжно** зафиксировать платёж и связать его с заказом, обеспечивая идемпотентность (повторный запрос не создаёт двойной оплаты).  

**ФТ-005:** Пользователь может отслеживать статус выполнения заказа и доставки в реальном времени.  

**ФТ-006:** Система отправляет уведомления (push/SMS) по ключевым событиям: подтверждение заказа, успешная/неуспешная оплата, назначение курьера, курьер в пути, доставка завершена, отмена/возврат.

## 2. Часть 2 — Нефункциональные требования (НФТ) с метриками

### НФТ-001: Задержка (Latency) — p50/p99
**Цели:** быстрый поиск/каталог и очень быстрый checkout; при этом платежи надёжны (внешний PSP может быть медленнее).

- **Поиск ресторанов/блюд (GET /search):**
  - p50 < **120 ms**
  - p99 < **300 ms**
- **Открытие ресторана/меню (GET /restaurants/{id}/menu):**
  - p50 < **150 ms**
  - p99 < **400 ms**
- **Операции с корзиной (POST/PUT /cart):**
  - p50 < **80 ms**
  - p99 < **200 ms**
- **Оформление заказа без оплаты (POST /orders):**
  - p50 < **200 ms**
  - p99 < **500 ms**
- **Подтверждение оплаты (POST /payments/confirm)** *(включая внешний PSP)*:
  - p50 < **900 ms**
  - p99 < **2500 ms**
- **Обновление статуса доставки (курьер/ресторан) (POST /orders/{id}/status):**
  - p50 < **60 ms**
  - p99 < **150 ms**
- **Получение статуса в приложении (GET /orders/{id}/track):**
  - p50 < **80 ms**
  - p99 < **200 ms**
- **Уведомления:**
  - постановка в очередь (enqueue): p99 < **100 ms**
  - доставка пользователю: **до 30 сек** (best-effort по каналу)

### НФТ-002: Пропускная способность (Throughput)
Проектные пики на API (без учёта CDN для медиа):
- **Search**: ≥ **10k RPS** (главная страница + фильтры в пик)
- **Menu/Restaurant page**: ≥ **6k RPS**
- **Cart ops**: ≥ **5k RPS**
- **Create order**: ≥ **500 RPS**
- **Payment confirm callbacks/confirm**: ≥ **200 RPS**
- **Status updates (курьер+ресторан)**: ≥ **2k RPS**

### НФТ-003: Доступность (Availability)
Разные домены — разная критичность.
- **Платежи (фиксация статуса платежа)**: **99.99%**/мес
- **Создание заказа**: **99.95%**/мес
- **Корзина**: **99.95%**/мес
- **Каталог/поиск**: **99.90%**/мес
- **Трекинг**: **99.90%**/мес
- **Уведомления**: **99.50%**/мес

### НФТ-004: Сохранность данных (Durability)
- **Платежи и заказы**:  
  - RPO = **0** для подтверждённых платежей и созданных заказов  
  - RTO ≤ **15 минут** для критичного контура  
  - Идемпотентность: все операции `create_order`, `confirm_payment` должны поддерживать **idempotency key**
- **Каталог/поиск/кэши**:  
  - допускается RPO до **15 минут** для поискового индекса/кэша (перестраиваемо)

### НФТ-005: Масштабируемость (Scalability)
- Рост нагрузки: **x2 MAU за 12–18 месяцев** без пересборки архитектуры.
- Горизонтальное масштабирование stateless API; отдельный read-optimized контур для поиска.
- Трекинг должен выдерживать рост числа активных доставок/соединений минимум **x3** в сезонные пики.

### НФТ-006: Надёжность платежей (Business-critical reliability)
- **Нельзя терять платежи**: платёжный статус должен быть согласован с заказом.
- События платежей должны быть обработаны **как минимум один раз** (at-least-once) + идемпотентность обработки.
- В случае деградации поиска/каталога **платежи и создание заказа должны продолжать работать** (изоляция контуров).

## 3. Часть 3 — Оценка нагрузки (Capacity Estimation)

Ниже — back-of-envelope. Берём средний MAU и закладываем запас на “обеденный пик” + сезонность.

### 3.1. Пользователи: MAU → DAU → пиковый онлайн

**Допущение A1:** MAU = **12 млн** (в пределах 5–20 млн).  
**Допущение A2:** DAU = **25% MAU** (у еды выше, чем у “редких” сервисов).  
→ DAU = 12,000,000 × 0.25 = **3,000,000**

**Сессии в день:** 1.3 сессии/пользователь/день  
→ 3,000,000 × 1.3 = **3,900,000 сессий/день**

**Средняя длительность сессии:** 5 минут  
“Минуты присутствия” в сутки: 3,900,000 × 5 = **19,500,000 минут**  
Средний онлайн: 19,500,000 / 1,440 ≈ **13,542** одновременных пользователей

**Обеденный пик:** коэффициент **×6** (еда сильно пикова)  
→ пиковый онлайн ≈ 13,542 × 6 ≈ **81,252** (~80k одновременных)

### 3.2. RPS: средний и пиковый (по операциям)

Опишем поведение 1 сессии.

**Допущение A3 (на 1 сессию):**
- search/list (главная + фильтры + выдача): **3** запроса
- открытие ресторана/меню: **2** запроса
- операции корзины (добавить/изменить): **4** запроса
- создание заказа: **0.25** (25% сессий доходят до “Оформить”)
- попытка оплаты: **0.20**

Сессий в секунду: 3,900,000 / 86,400 ≈ **45.14 сессий/сек**

**Средний RPS:**
- search: 45.14 × 3 ≈ **135.4 RPS**
- menu: 45.14 × 2 ≈ **90.3 RPS**
- cart: 45.14 × 4 ≈ **180.6 RPS**
- create order: 45.14 × 0.25 ≈ **11.3 RPS**
- payment: 45.14 × 0.20 ≈ **9.0 RPS**

**Пиковый RPS (обеденный пик):** трафик концентрируется сильнее среднего, берём коэффициент **×12** к среднесуточному.  
- search: 135.4 × 12 ≈ **1,625 RPS**
- menu: 90.3 × 12 ≈ **1,084 RPS**
- cart: 180.6 × 12 ≈ **2,167 RPS**
- create order: 11.3 × 12 ≈ **136 RPS**
- payment: 9.0 × 12 ≈ **108 RPS**


Почему тогда в НФТ-002 “проектные пики” выше (10k RPS на search)?
- потому что MAU может быть 20 млн,
- бывают акции/плохая погода/локальные всплески,
- выдача перезапрашивается при каждом действии в фильтрах.  
Инженерно проектируем с запасом **~x5** к “среднему кейсу”.

### 3.3. Пиковые активные заказы

Оценим количество “живых” заказов в пике.

**Допущение A4:** в день оформляется **2% от DAU** заказов  
→ 3,000,000 × 0.02 = **60,000 заказов/день**

**Средняя длительность жизненного цикла заказа:** 45 минут  
Заказы в минуту: 60,000 / 1,440 ≈ **41.7 заказ/мин**  
Среднее число активных заказов: 41.7 × 45 ≈ **1,875 активных**  
Обеденный пик ×6 → **~11,250 активных заказов одновременно**


### 3.4. Latency Budget для критичного сценария (checkout: create order + payment)

Критичный путь: **“Оформление заказа и фиксация оплаты”**.

SLO: **end-to-end p99 < 2.7 сек** на сценарий “Place order + Pay” (включая PSP).

Разложение latency budget:

1) **Client → API Gateway**: 20 ms  
2) **Order Service (создание заказа)**:  
   - бизнес-логика: 60 ms  
   - запись в БД: 180 ms  
   - публикация события (outbox → broker): 40 ms  
   Итого: **280 ms**
3) **Payment Service → PSP (эквайринг/3DS)**: p99 **до 2,000 ms**  
4) **Обработка callback/webhook PSP + финализация статуса**:  
   - приём webhook: 80 ms  
   - транзакция обновления платежа+заказа: 200 ms  
   Итого: **280 ms**
5) **UI обновление / подтверждение**: 100 ms

Итого p99: 20 + 280 + 2000 + 280 + 100 = **2,680 ms**

Бутылочное горлышко — внешний PSP/3DS. Способы уложиться:
- UI `processing` + финал асинхронно,
- оптимизация платежного флоу (сохранённые карты, hosted page),
- параллелизм шагов без нарушения целостности платежей.

### 3.5. Хранилище: объём данных через 1 год и 3 года

#### Рестораны и меню
**Допущение A5:** 200,000 ресторанов, 80 блюд/ресторан.  
- Ресторан (метаданные): ~1 KB → **~200 MB**
- Блюдо: ~0.6 KB → 200,000 × 80 × 0.6 KB ≈ **9.6 GB**
- Картинки блюд: 3 фото/блюдо, 120 KB  
  → 200,000 × 80 × 3 × 120 KB ≈ **5.76 TB** (object storage + CDN)

#### Заказы и платежи
**Допущение A6:** 60,000 заказов/день.  
За год: **21.9 млн**; за 3 года: **65.7 млн**.

- Заказ: ~1.2 KB  
  - 1 год: **~26.3 GB**
  - 3 года: **~78.8 GB**
- Платёж: ~0.8 KB  
  - 1 год: **~17.5 GB**
  - 3 года: **~52.6 GB**
- Статусные события: 10/заказ × 0.2 KB  
  - 1 год: **~43.8 GB**
  - 3 года: **~131 GB**

Итого ядро в БД (без индексов/реплик) за 3 года: порядка **~260 GB**.  
С индексами и репликацией — порядка **~1–2 TB**.


### 3.6. Bandwidth: входящий/исходящий трафик

**Допущение A7 (размеры ответов):**
- search: ~40 KB
- menu: ~60 KB
- cart: ~5 KB

Проектные пики (НФТ-002):
- search: 10k RPS × 40 KB ≈ **3.2 Gbps**
- menu: 6k RPS × 60 KB ≈ **2.9 Gbps**
- cart: 5k RPS × 5 KB ≈ **0.2 Gbps**

Итого API egress: **~6.4 Gbps**.  
Медиа — только через CDN.

## 4. Допущения, ограничения, открытые вопросы, противоречия

### 4.1. Допущения (Assumptions)
**A1:** MAU=12 млн, DAU=25% MAU.  
**A2:** 1.3 сессии/пользователь/день, 5 минут/сессия.  
**A3:** Профиль запросов: search×3, menu×2, cart×4, create×0.25, pay×0.20.  
**A4:** 2% DAU делают заказ/день; жизненный цикл заказа ~45 минут.  
**A5:** 200k ресторанов, 80 блюд/ресторан, 3 фото/блюдо.    
**A6:** Изображения через CDN.

### 4.2. Ограничения (Constraints)
- Ограниченный бюджет → избегаем сверхсложной мультизонной записи.
- Маленькая команда → простая реализуемая архитектура.
- Внешний PSP влияет на latency → UX учитывает асинхронность.

### 4.3. Открытые вопросы (Questions)
1) Обязательные способы оплаты (карта/СБП)?  
2) Hold vs charge сразу? Частичные возвраты?  
3) SLA “ничего не теряется” — только платежи или и заказы?  
4) Статусы: кто выставляет, какие исключения нужны.  
5) Отмена после начала готовки: правила компенсаций.  
6) Требования к модерации меню/фото.  
7) География/мульти-регион для СНГ.  
8) Политика хранения адресов/PII.  
9) Нужна ли история цен/акций и аудит изменений меню.

### 4.4. Противоречия и решения

**P1:** “Оплата — самое важное, ничего не теряется” vs “checkout должен быть очень быстрым”.  
**Решение:**  
- Разделение контуров: read-path (поиск/меню) — кэш/индекс; write-path (заказ/платёж) — транзакции + outbox + идемпотентность.  
- UI: `processing` при медленном PSP, финал статуса асинхронно.  
- При деградации: ограничиваем тяжёлые фичи поиска, но сохраняем оплату/заказы.

## 5. Приложение — статусы

### 5.1. Order Status
`created → accepted → cooking → ready → courier_assigned → on_the_way → delivered`  
Исключения: `cancelled`, `payment_failed`, `refunding`, `refunded`.

### 5.2. Payment Status
`created → processing → succeeded / failed`  
Возвраты: `refunding → refunded`  
Идемпотентность по `payment_intent_id` / `idempotency_key`.
